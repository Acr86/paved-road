# 0009. Preview environments via ApplicationSet matrix with a TTL and orphan janitor

Date: 2026-06

## Status

Accepted

## Context

Reviewers need to click through a change, not just read it. That means an environment per pull
request, created without asking the platform team and destroyed without anyone remembering to.
The failure mode that kills preview systems is not creation — any CI job can `kubectl apply` —
it is reclamation: environments that outlive their PR accumulate until the cluster is a
graveyard and someone declares preview environments "too expensive".

Two forces shape the design: previews must be declarative (drift-corrected, pruned when the
source disappears), and cleanup must be safe enough to run unattended (a janitor that can
mass-delete on a bad day is worse than no janitor).

## Decision

Previews are generated by an
[ApplicationSet matrix](../../deploy/argocd/apps/applicationset-previews.yaml): a pull-request
generator (open PRs labelled `preview` on Acr86/paved-road) crossed with a git directory
generator over `deploy/kustomize/services/*/overlays/preview`. Each open labelled PR therefore
gets one Application per service — a complete environment — in namespace `pr-<N>`, with the
image overridden to `ghcr.io/acr86/paved-road/<svc>:pr-<N>` (built by `preview.yml` on label) and
the ingress host patched to `pr-<N>.<svc>.127.0.0.1.nip.io`. Closing or unlabelling the PR
prunes the Applications.

ArgoCD does not delete namespaces it created, so reclamation is a janitor: a CronJob in
`platform-system` running every 15 minutes, executing `platform preview gc --orphans` from the
same container image as the CLI — the decision logic in
[`preview.py`](../../platform-cli/src/platform_cli/preview.py) is the unit-tested code, not a
shell reimplementation of it. Two independent rules reclaim a namespace:

- **TTL**: label `pavedroad.dev/expires-at` (absolute unix epoch) in the past.
- **Orphan**: a preview namespace whose destination no longer has any ArgoCD Application
  (PR closed) and which is older than a 30-minute grace period.

The safety properties are the unit-tested part: a malformed `expires-at` is reported, never
deleted; an unknown creationTimestamp is never reaped; namespaces not matching `pr-*` or
missing the preview label are never touched; and the grace period means a transient ArgoCD
outage (an empty Application list) cannot mass-delete fresh environments.

The cost story follows: nothing survives its PR by more than one janitor interval plus grace.

## Honest limitations

- The git directory generator lists overlays at `main`, so a service added in the same PR does
  not get a preview Application until that PR merges. Existing services preview the PR's
  revision; brand-new ones do not.
- The GitHub API is polled anonymously, so an environment can take up to ~5 minutes to appear
  after labelling. Acceptable for a public reference repo; a token tightens it.
- Every service is previewed for every PR — there are no change-scoped previews yet. The
  natural next step is paths-filter-scoped previews, generating Applications only for services
  the PR actually touches.

## Alternatives considered

**Namespace-per-PR via imperative CI kubectl.** Simpler to start: the PR pipeline applies
manifests, a close hook deletes them. Rejected because there is no reconciliation or pruning —
a failed cleanup job, a force-pushed branch, or a manifest renamed mid-PR leaves drift that
nothing converges, and the close hook is exactly the "someone must remember" step this design
exists to remove.

**vcluster-per-PR.** Stronger isolation: each PR gets a virtual control plane. Rejected as
heavier than the trust model warrants — PRs here come from the repo's own contributors, not
anonymous third parties, and the per-preview overhead of a control plane buys isolation we do
not currently need.

## Consequences

- Previews share the cluster. A hostile or runaway PR workload can starve neighbors —
  including other previews and platform components. Resource requests and limits on every
  rendered workload are the current containment; per-namespace ResourceQuotas are the future
  one. This is an accepted risk, and it is the first thing to revisit if PR authorship ever
  extends beyond trusted contributors.
- Reclamation correctness depends on the janitor CronJob itself being healthy; the
  `PreviewJanitorFailing` alert exists because a silently dead janitor recreates the graveyard
  problem on a delay.
